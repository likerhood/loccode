对，你这个提醒很关键：**我们的方法必须是 LLM-based Agent**，否则它就会变成一个“IR + 规则检索器”，创新性和故事都弱。更合适的范式不是“LLM 随便 ReAct 搜索”，也不是“完全 agentless 的固定流水线”，而是：

> **证据约束的诊断型 LLM Agent：LLM 负责理解症状、选择诊断动作、解释机制链；工具负责可靠检索、局部展开、校验和剪枝。**

这个定位会比“多模态证据图”更清楚，也更符合顶会论文的逻辑闭环。

**从相关工作看范式应该怎么立**
SWE-agent 的核心启发是：Agent 能力很大程度来自 **Agent-Computer Interface**，也就是工具界面如何设计会影响 agent 行为，而不是单纯模型强弱。它强调让 agent 能导航仓库、操作文件、运行反馈。这个对我们很重要：我们不是只写 prompt，而是设计一个面向定位任务的诊断接口。来源：SWE-agent 论文。  
Agentless 的启发相反：复杂 agent 未必总是好，定位、修复、验证这种分阶段流程反而稳定；它的 localization 是分层的，先文件，再类/函数，再细粒度位置。来源：Agentless。  
CoSIL 的启发是：LLM 上下文放不下全仓库，所以要动态搜索代码图、控制上下文、剪枝搜索方向。来源：CoSIL。  
GALA 的启发是：多模态 bug localization 不能只把图片转文字，要显式建视觉图和代码图，并做层级对齐。但它的问题是图对齐偏静态、偏结构相关，不一定能形成“症状如何导致代码根因”的诊断闭环。来源：GALA。

所以我们的范式可以总结成：

```text
Agentless 给我们阶段稳定性
SWE-agent 给我们工具接口思想
CoSIL 给我们动态局部图和剪枝
GALA 给我们多模态结构化证据
我们把它们合成：证据驱动的诊断型定位 Agent
```

**方法名字可以这样讲**
我建议叫：

> **Evidence-Guided Diagnostic Agent for Multilingual and Multimodal Issue Localization**

中文可以叫：

> **多源证据驱动的诊断型代码定位 Agent**

或者更专利化一点：

> **一种基于多源证据路由与局部机制图推理的软件缺陷定位方法**

这里的关键词是：

- 多源证据，不局限图片。
- 诊断型 Agent，不是普通 ReAct。
- 局部机制图，不是全仓库大图。
- 证据路由，不是自由搜索。
- 可验证三层输出，file/module/function/line。

**整体逻辑闭环**
完整流程应该是这个：

```text
输入 Issue / 图片 / URL / 复现代码 / 错误日志
        ↓
1. Evidence Agent：抽取结构化证据
        ↓
2. Recall Agent：MM-IR 召回候选池
        ↓
3. Diagnostic Planner：选择诊断动作
        ↓
4. Tool Executor：执行局部搜索/切片/图扩展
        ↓
5. Mechanism Graph Builder：构造局部机制图
        ↓
6. Critic / Verifier：剪枝、校验、反证
        ↓
7. Ranker Agent：输出 file/module/function/line + evidence chain
```

注意，这里面不是一个 LLM 从头到尾乱搜。它是一个 **受约束的 agent**：

```text
LLM 决策：
- 这个 issue 属于什么症状？
- 哪些证据重要？
- 应该触发哪个诊断动作？
- 哪条机制链更可信？

工具保证：
- 文件是否存在
- 函数是否存在
- 字符串是否精确命中
- 变量是否有定义/使用链
- 候选是否在局部图范围内
- 输出 schema 是否可解析
```

这样才像顶会方法：既有 agent 的推理能力，又有系统机制约束。

**第一阶段：Evidence Agent**
这一步 LLM 不是直接定位代码，而是把 issue 转成结构化证据。

输出类似：

```json
{
  "evidence": [
    {
      "type": "URL_PATH",
      "text": "/jetpack/connect/plans",
      "confidence": 0.95,
      "suggested_actions": ["anchor_binding_search"]
    },
    {
      "type": "ROLE_OR_STATE",
      "text": "subscriber user",
      "confidence": 0.85,
      "suggested_actions": ["state_dataflow_trace"]
    },
    {
      "type": "VISUAL_SYMPTOM",
      "text": "site entry screen shown instead of plans page",
      "confidence": 0.80,
      "suggested_actions": ["visual_mechanism_probe"]
    }
  ]
}
```

这一步的创新点是：**多模态内容不是转成自然语言描述就结束，而是转成可执行诊断动作。**

图片/OCR、URL、错误日志、复现代码、配置文本，都变成 typed evidence。

如果样本没有图片，也没关系。比如 OmniGIRL 里很多只有代码块：

```text
for (await of [1,2,3])
```

Evidence Agent 会抽成：

```json
{
  "type": "REPRO_CODE",
  "concept": "parser_context",
  "actions": ["parser_context_search"]
}
```

所以这个方法不是“有图才工作”，而是“有任何证据都能路由”。

**第二阶段：Recall Agent**
这里学习 MM-IR 和 LocAgent。

但不是每轮都让 LLM 自由调用 `search_code_snippets`。我们先建统一候选池：

```text
File candidates: top-40
Module candidates: top-80
Function candidates: top-100
String/path/config candidates: top-100
```

召回来源：

```text
BM25 / dense retrieval / symbol index / string index / route index / config index
```

它的作用是缩小搜索空间：

```text
全仓库几千到几十万个实体
        ↓
几十个文件 + 一百个函数/字符串候选
```

这个阶段回答你说的“不搜索全部仓库”：  
我们不是在全仓库跑图搜索，而是在离线索引上做 top-K 召回，之后所有局部图扩展都被限制在候选池及其邻域内。

**第三阶段：Diagnostic Planner**
这是 LLM Agent 的核心。

它根据证据选择动作，而不是四个动作全跑：

```text
URL / path / 字符串      -> Anchor Binding Search
null / undefined / stack -> State/Dataflow Trace
视觉裁剪 / 颜色 / 布局   -> Visual Mechanism Probe
复现代码 / config        -> Repro/Config Context Search
```

可以让 LLM 输出计划：

```json
{
  "selected_actions": [
    {
      "name": "anchor_binding_search",
      "reason": "URL path /jetpack/connect/plans is a strong route anchor",
      "budget": 8
    },
    {
      "name": "state_dataflow_trace",
      "reason": "subscriber role changes redirect decision",
      "budget": 6
    }
  ],
  "skipped_actions": [
    {
      "name": "visual_mechanism_probe",
      "reason": "screenshot explains symptom but route/auth evidence is stronger"
    }
  ]
}
```

这一步很重要，因为它让 agent 有“诊断策略”，不是瞎搜。

**第四阶段：Tool Executor**
每个动作对应固定工具组合。

例如 `Anchor Binding Search`：

```text
search route/path literal
search string literal
search file/path basename
expand imports/callers depth=1
```

例如 `State/Dataflow Trace`：

```text
search property access
search selector/state variable
find definitions
backward slice one or two hops
```

例如 `Visual Mechanism Probe`：

```text
visual symptom -> mechanism concept
clipped -> clip/chartArea/layout/boundary
color alpha -> blend/opacity/fillStyle/shader
overlap -> layout/position/z-index/measure
```

例如 `Repro/Config Context Search`：

```text
repro token -> parser/tokenizer/test fixture
config option -> config loader/plugin/rule resolver
```

这里 LLM 不直接“看全仓库”，而是通过工具拿局部证据。

**第五阶段：局部机制图**
这一步是我们区别于普通 agent 的关键。

普通 agent 轨迹是：

```text
我搜了 A
我看了 B
我觉得 C
```

我们的机制图是：

```text
EvidenceNode -> ActionNode -> CodeNode -> FlowNode -> ConceptNode -> EvidenceNode
```

比如 Jetpack Connect：

```text
URL_PATH: /jetpack/connect/plans
  -> triggers Anchor Binding Search
  -> CodeNode: authorize.js::redirect
  -> reads FlowNode: scope
  -> ConceptNode: ROLE_SCOPE
  -> explains EvidenceNode: subscriber should redirect wp-admin
```

如果某个候选是 `site-url-input.jsx`：

```text
VISUAL_TEXT: Connect a self-hosted WordPress
  -> matches CodeNode: site-url-input.jsx
```

它只覆盖了视觉文本，没有解释 subscriber、scope、redirect loop，所以分数低。

这就是“机制图”的意义：**不是谁像 issue，而是谁能解释 issue。**

**第六阶段：Critic / Verifier**
这个部分借鉴 CoSIL 的 prune，但更工程化。

CoSIL 是拿到代码后问一个小 agent：

```text
这个函数/类是否和 bug 相关？
True / False
```

我们可以做成三层：

1. 规则 verifier：

```text
文件存在？
函数存在？
line 在函数范围内？
字符串是否精确出现？
变量是否有定义/使用？
```

2. 证据链 verifier：

```text
候选是否至少覆盖一个强证据？
是否形成 Evidence -> Action -> Code -> Mechanism 的路径？
是否只是 surface match？
```

3. LLM critic：

只在难判断时调用：

```text
给定证据链和代码片段，判断它是否解释症状。
```

这样成本比 CoSIL 小，不会每个候选都问 LLM。

**第七阶段：最终输出**
必须强 schema：

```json
{
  "file": "client/jetpack-connect/authorize.js",
  "module": "JetpackAuthorize",
  "function": "redirect",
  "line": 165,
  "confidence": 0.84,
  "evidence_chain": [
    "URL path /jetpack/connect/plans",
    "subscriber role in issue",
    "scope is parsed into role",
    "redirect() controls wp-admin return"
  ]
}
```

这个解决我们之前反复遇到的问题：

- raw output 空。
- 只输出文件。
- function/module 丢。
- 结果不可解析。
- LLM finish 但没内容。

**创新点应该这样写**
我觉得可以凝练成三个创新点。

**创新点 1：证据约束的诊断动作路由**

现有 LLM agent 常用 ReAct：模型自己决定搜什么。问题是容易发散、重复、搜错方向。  
我们把 issue 中的多源证据映射成动作：

```text
URL -> route/string anchor
error -> dataflow trace
visual symptom -> visual mechanism probe
repro code -> parser/config context
```

这不是 prompt trick，而是一个可实现的 routing policy。

**创新点 2：候选池约束的局部机制图**

GraphLocator/GALA 类方法容易图太大、太静态。  
我们不是建全仓库图，而是：

```text
MM-IR top-K 候选
+ action-specific local expansion
+ depth/budget pruning
= issue-specific local mechanism graph
```

每个 issue 的图只有几十到几百个节点。

**创新点 3：证据链可验证的三层定位输出**

现有 agent 最终输出不稳定。  
我们要求每个预测必须带：

```text
file/module/function/line
evidence_chain
confidence
```

并用 resolver 校验。  
这使定位结果可评估、可解释、可修复。

**为什么这个比现有 baseline 更有逻辑**
和 MM-IR 比：  
MM-IR 是召回器，能找文本相似，但不能判断根因。我们把它作为候选池，再用机制图做解释性重排。

和 LocAgent 比：  
LocAgent 有工具搜索，但 query 主要由 LLM 自由生成。我们用证据路由约束 query，减少空转和发散。

和 CoSIL 比：  
CoSIL 动态图搜索强，但主要围绕代码调用/模块关系。我们把多模态、URL、配置、错误、复现代码都变成诊断动作，图不只是 call graph。

和 GraphLocator 比：  
GraphLocator 依赖重图，遇到混合语言、大仓库、特殊路径会慢和崩。我们只做候选池邻域的局部机制图。

和 GALA 比：  
GALA 强在视觉图和代码图对齐，但对齐容易停在“视觉元素对应哪个组件”。我们进一步问：这个代码实体是否解释了行为根因？例如截图里的按钮可能对应 UI 文件，但 redirect bug 根因在 auth/scope 逻辑。

**一个可以写进论文/专利的范式图**
可以这样画：

```text
                 ┌────────────────────────┐
                 │ Issue / Image / URL /  │
                 │ Repro / Error / Config │
                 └───────────┬────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ Evidence Agent          │
                 │ typed evidence nodes    │
                 └───────────┬────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ MM-IR Candidate Recall  │
                 │ file/module/function    │
                 └───────────┬────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ Diagnostic Planner LLM  │
                 │ choose actions + budget │
                 └───────────┬────────────┘
                             ↓
          ┌──────────────────┼──────────────────┐
          ↓                  ↓                  ↓
  Anchor Search       Dataflow Trace      Visual/Repro Probe
          └──────────────────┬──────────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ Local Mechanism Graph   │
                 │ evidence-action-code    │
                 └───────────┬────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ Critic + Verifier       │
                 │ prune / validate        │
                 └───────────┬────────────┘
                             ↓
                 ┌────────────────────────┐
                 │ Ranked Localization     │
                 │ file/module/function    │
                 └────────────────────────┘
```

**一句话版本**
如果要一句话概括这个范式：

> 我们提出一种证据约束的诊断型 LLM Agent：它先用 MM-IR 缩小候选空间，再把 issue 中的文本、视觉、URL、错误、复现代码和配置证据转化为可执行诊断动作，按需构建局部机制图，并通过证据链校验输出 file/module/function/line 级定位结果。

这个说法会比“多模态证据图”更稳，也更像一个完整方法：有输入、有 agent 决策、有工具、有搜索、有剪枝、有解释、有输出闭环。
