我重新看了根目录几个文档和结果汇总后，觉得我们的创新点应该从“做一个更大的图”调整成：

> **做一个证据驱动、候选受限、可验证输出的诊断型 LLM Agent。**

也就是 **ECDA：Evidence-Convergent Diagnostic Agent，多源证据汇合的诊断型代码定位 Agent**。

它不是单纯针对“Graph 太重”，而是同时针对这几类真实失败：

1. 多语言仓库导致 parser/图构建/实体抽取不稳。
2. 多模态/多源 issue 证据没有被结构化使用。
3. LLM Agent 自由搜索容易跑偏、空输出、只给 file。
4. IR/BM25 召回强，但不能解释根因。
5. Graph 方法太重，且对特殊路径、混合文件类型敏感。
6. GALA 虽有视觉图，但最终 module/function 闭环不稳定。

**我们从 baseline 结果看到的问题**
先把本地结果里的核心现象压缩一下。

| 方法 | 真实表现 | 暴露的问题 |
|---|---|---|
| BM25-MMIR | SWE 60 File Acc@15 66.67，Function Acc@15 43.33，很强 | 文本召回强，但只是“像不像”，不能解释根因，PRE/F1 偏低 |
| LocAgent | SWE 60 三层整体较强，LocAgent Omni 48 File Acc@15 47.92 | 工具搜索依赖 LLM，自由 query 容易漂；曾出现 finish 空、file-only、实体失效 |
| CoSIL | 有预选、prune、函数级思想 | 多语言/多模态不是核心，图和工具更偏 Python/代码调用关系 |
| GALA | SWE 文件级 Acc@15 61.67 | 当前产物 `final_modules/final_functions` 为空，视觉到函数级闭环没落到评估输出 |
| GraphLocator | SWE File Acc@15 28.33，Omni 空预测高 | 全仓库图慢、重；混合语言/特殊路径会崩；Omni module/function 空预测高 |
| MM-IR full | SWE full File Acc@15 65.69，Omni 631 File Acc@15 51.82 | 适合做候选池，但缺少诊断性重排 |

所以这不是“某个 baseline 写坏了”，而是说明多语言多模态定位本身需要一个更稳的范式：

```text
先召回，再诊断；
先证据，再动作；
先局部，再扩展；
先验证，再输出。
```

**外部工作给我们的逻辑**
联网查的相关工作可以支撑这个范式：

- **SWE-agent** 说明 Agent 能力依赖合适的 Agent-Computer Interface，不只是模型强弱。我们也要设计面向定位的诊断接口，而不是让 LLM 自由搜。  
  https://arxiv.org/abs/2405.15793

- **Agentless** 说明简单分阶段流程在软件工程任务里很稳：localization、repair、validation。我们也保留分阶段稳定性。  
  https://arxiv.org/abs/2407.01489

- **IRCoT** 说明多步问题里“一次检索”不够，下一步检索依赖已经推理出的中间结论。我们的证据动作路由就是 issue localization 版 IRCoT。  
  https://aclanthology.org/2023.acl-long.557/

- **EfficientRAG** 说明多跳检索要高效，不能每一步都靠大模型暴力调用。我们的局部机制图和预算剪枝正好对应这个问题。  
  https://aclanthology.org/2024.emnlp-main.199/

- **CoSIL** 说明代码定位需要动态搜索和剪枝，但它主要是代码图/调用图。  
  https://arxiv.org/abs/2503.22424

- **GALA** 说明多模态定位不能只把图片转文字，要结构化视觉证据；但我们还要把视觉证据继续绑定到代码根因机制。  
  https://arxiv.org/abs/2604.08089

- **ARISE** 说明函数/行级定位需要 data-flow slicing，结构图只说明相关，不能说明值如何传播。  
  https://arxiv.org/abs/2605.03117

**我们的核心创新点**
我建议最终写成 4 个创新点。

**创新点 1：多源证据到诊断动作的路由**
现有方法常见两种：

```text
BM25/MM-IR：把所有文本拼成 query
LocAgent：让 LLM 自己决定搜什么
```

我们改成：

```text
EvidenceNode -> DiagnosticAction
```

证据包括：

```text
Issue 文本
截图/OCR
URL
复现代码
错误日志
配置文本
路径/route
```

不同证据触发不同动作：

| 证据 | 动作 | 解决的问题 |
|---|---|---|
| URL/path/string/i18n | Anchor Binding Search | 防止只看截图表面 |
| null/undefined/stack | State/Dataflow Trace | 找状态来源，不只找报错页面 |
| clipped/color/layout/render | Visual Mechanism Probe | 把视觉症状转成渲染机制 |
| repro code/config | Repro/Config Context Search | 处理 OmniGIRL 大量无图但有复现代码的样本 |

这个点比“OCR 拼 prompt”高级，因为它把多源证据变成 **可执行诊断动作**。

**创新点 2：多语言统一代码实体底座**
GraphLocator/LocAgent/CoSIL 都暴露过多语言问题：JS/TS/Java/Python 混合时，实体抽取、图构建、工具返回不稳定。

我们应该设计统一实体层：

```json
{
  "entity_id": "repo@commit:file:function:start:end",
  "language": "JavaScript",
  "file": "client/jetpack-connect/authorize.js",
  "kind": "function",
  "name": "redirect",
  "start_line": 165,
  "end_line": 205,
  "relations": ["contains", "calls", "imports", "reads_state"]
}
```

语言适配器只负责把不同语言转成统一实体：

```text
Python -> class/function/method/import/call
JS/TS -> function/class/component/export/import/hook/selector
Java -> class/method/package/import/call
Config -> option/plugin/rule/loader
```

这解决的是：**真实仓库不是单语言代码图，而是混合语言、配置、测试、构建文件共同构成的问题空间。**

**创新点 3：候选受限的局部机制图，而不是全仓库图**
GraphLocator 慢和崩的根因是图太重，全仓库构建和遍历成本高。

我们改成：

```text
MM-IR 先召回 top-K 候选
只围绕候选池构建 issue-specific local graph
每轮只扩展 1~2 跳
```

图节点：

```text
EvidenceNode：issue/OCR/URL/error/repro/config
ConceptNode：AUTH_REDIRECT / NULL_STATE / LAYOUT_CLIP / PARSER_CONTEXT
ActionNode：anchor_search / dataflow_trace / visual_probe / config_search
CodeNode：file/module/function/line
FlowNode：variable/state/config/token
```

这张图不是 repo graph，而是“案情图”：

```text
证据 -> 动作 -> 代码候选 -> 局部关系 -> 机制解释
```

它解决的是：**既保留图推理能力，又避免全仓库 skeleton graph 的笨重和脆弱。**

**创新点 4：证据汇合排序与 schema verifier**
BM25/MM-IR 能找相似文件，但不能判断根因。我们的排序看“多条证据是否汇合”。

```text
Score(c) =
  MMIR_score
+ EvidenceCoverage
+ ActionAgreement
+ MechanismPath
+ FlowSupport
+ CrossEvidenceCentrality
- SurfaceOnlyPenalty
- ExpansionCost
- HubPenalty
```

直观解释：

- 多条证据都指向同一个函数，分高。
- URL 和 dataflow 在一个文件汇合，分高。
- 截图文字只命中 UI 表层，但不能解释行为，扣分。
- `index.js`、`utils.js` 这种高频 hub，扣分。
- 路径太远、扩展太多，扣分。

最终输出必须是：

```json
{
  "file": "...",
  "module": "...",
  "function": "...",
  "line": 123,
  "evidence_chain": [
    "URL /jetpack/connect/plans",
    "subscriber role",
    "scope -> role",
    "redirect() controls wp-admin return"
  ],
  "confidence": 0.84
}
```

这直接解决 LocAgent/GALA/GraphLocator 的三层输出不稳定问题。

**结合具体案例看我们解决了什么**
**案例 1：Jetpack Connect**
真实修改：

```text
client/jetpack-connect/authorize.js
client/jetpack-connect/utils.js
```

Baseline 容易偏到：

```text
site-url-input.jsx
plans.jsx
```

因为截图里是 site entry 页面。

我们的图：

```text
URL /jetpack/connect/plans
  -> Anchor Search
  -> authorize.js::redirect

subscriber user
  -> ROLE_SCOPE
  -> utils.js::getRoleFromScope

expected wp-admin redirect
  -> AUTH_REDIRECT
  -> authorize.js::redirect
```

汇合点：

```text
utils.js::getRoleFromScope -> role
authorize.js::redirect -> redirect decision
```

所以能压过 `site-url-input.jsx`。

**案例 2：Help Contact Form**
真实修改：

```text
help-contact-form/index.jsx
state/help/selectors.js
```

问题：

```text
Cannot read property 'ID' of null
no primary site
```

传统检索容易找 contact-form UI，但 selector 不稳定。

我们的路径：

```text
ERROR: Cannot read property ID of null
  -> State/Dataflow Trace
  -> selectedSite.ID
  -> help-contact-form/index.jsx

TEXT: no primary site
  -> NULL_STATE
  -> getHelpSelectedSite
  -> state/help/selectors.js
```

两个文件通过 `selectedSite` flow 汇合。

**案例 3：Chart.js clipped**
问题：

```text
rightmost point clipped
tooltip still appears
```

普通视觉/文本检索可能找 chart UI 或 tooltip。

我们的机制：

```text
rightmost point clipped
  -> LAYOUT_CLIP
  -> chartArea / clip / radius / draw

tooltip still appears
  -> DATA_EXISTS
  -> rules_out dataset parsing
```

这里有反证据：tooltip 正常说明数据存在，应该降低 data parsing，提升 layout/clip/render。

**案例 4：Babel parser**
OmniGIRL 很多样本没有图，但有复现代码/URL。

```text
for (await of [1,2,3])
Babel REPL URL
parser rejects
```

我们的动作：

```text
REPRO_CODE -> PARSER_CONTEXT
URL -> external repro parser
ERROR -> parser/token context
```

候选汇合在：

```text
parseFor / parseForIn / await token context / tokenizer
```

这说明我们的多源证据框架不是只服务截图，而是覆盖多语言 issue。

**针对子集和全量的实验规划**
为了证明这个范式有效，建议按阶段做。

**阶段 A：ECDA-BM25 轻量版**
用现有 BM25-MMIR 候选池，不引入 dense 模型。

比较：

```text
BM25-MMIR
BM25-MMIR + Evidence Router
BM25-MMIR + Evidence Router + Local Mechanism Graph
BM25-MMIR + Evidence Router + Local Graph + Verifier
```

跑：

```text
SWE-bench Multimodal 60
OmniGIRL 60
SWE full-candidates 102
Omni full-candidates 631
```

预期提升点：

```text
Function Acc@K
Module Acc@K
Strict SL
MRR
空输出率
```

**阶段 B：多语言实体底座消融**
对比：

```text
file-only
file + module/function
file + module/function + string/config/route
file + module/function + dataflow/config flow
```

看是否解决：

```text
GraphLocator module/function 空
GALA module/function N/A
LocAgent file-only 输出
```

**阶段 C：证据动作消融**
分别关掉：

```text
无 Anchor Search
无 Dataflow Trace
无 Visual Probe
无 Repro/Config Search
无 Bridge Action
无 SurfaceOnlyPenalty
```

用案例验证：

```text
Jetpack -> Anchor/Auth 有用
Help Form -> Dataflow 有用
Chart.js/p5.js -> Visual Mechanism 有用
Babel/Tailwind/Webpack -> Repro/Config 有用
```

**阶段 D：效率实验**
对比 GraphLocator：

```text
全仓库 graph 构建时间
每样本平均工具调用数
局部图节点数
平均定位耗时
崩溃率
```

目标证明：

```text
不是图越大越好，而是局部机制图更稳、更轻。
```

**我们最终针对的问题**
可以把“针对什么解决什么”写成专利/论文式表述：

| 现有问题 | 我们的设计 | 解决方式 |
|---|---|---|
| 图方法太重 | 候选池受限局部机制图 | 只扩展 top-K 邻域，不全仓库 BFS |
| 多语言仓库结构不统一 | 统一 CodeNode 实体模型 | JS/TS/Python/Java/config 统一到 file/module/function/config |
| 多模态证据停在文本描述 | EvidenceNode + DiagnosticAction | 图片/OCR/URL/错误/代码块变成动作 |
| BM25 只能词面匹配 | Evidence convergence scoring | 通过证据汇合和机制路径重排 |
| Agent 搜索不稳定 | Evidence Router + tool budget | 限定动作和搜索空间 |
| 输出不可评估 | Schema Verifier | 强制 file/module/function/line/evidence_chain |
| GALA 停在视觉对齐 | MechanismPath + SurfaceOnlyPenalty | 视觉表层文件若不能解释行为则降权 |
| CoSIL 代码图偏单一 | 多源证据 + config/dataflow/visual actions | 不只 call graph，还包括状态、配置、视觉机制 |
| GraphLocator 崩溃/慢 | Lazy local graph + robust indexes | 特殊路径只影响局部 action，不影响全流程 |

**一句话创新点**
可以这样写：

> 我们提出一种多源证据汇合的诊断型 LLM Agent。该方法先使用轻量 MM-IR 建立候选代码池，再将 issue 中的文本、截图、URL、错误日志、复现代码和配置文本解析为 typed evidence，并由 LLM Planner 将证据路由到可执行诊断动作；随后仅在候选池邻域内按需构建局部机制图，通过多条证据路径在代码实体上的汇合程度、机制解释能力和 schema 校验，对 file/module/function/line 进行可解释排序。

这个创新点不只是“多模态”或“多语言适配”，而是完整范式：

```text
多源证据 -> 诊断动作 -> 局部机制图 -> 证据汇合 -> 可验证三层定位
```

它正好回应了我们跑 baseline 时看到的所有主要问题。
