# 三层定位评估样本清洗与可评估性分析

本文专门分析一个评估口径问题：我们当前三层定位指标包含 `File / Module / Function`，并且排名指标主要看 `Acc@1` 到 `Acc@15`。如果某个样本真实需要修改的 file、module 或 function 数量超过 15，那么严格 full-coverage 的 `Acc@15` 天然不可能为 1；如果某个样本虽然有真实修改文件，但真实修改行无法映射到 module/function，那么这个样本用于 module/function 指标也会混入结构抽取噪声。

所以这里不讨论模型好坏，而是先回答：**哪些样本适合作为三层 Acc@15 的主评估集？哪些样本应该作为长尾诊断集或 file-only 集合单独报告？**

数据来源与 `006_benchmark样本URL图片语言和Gold分布分析.md` 相同，module/function gold 由 gold patch 的修改行映射到 `repo_structures` 中的代码实体得到。

## 1. 为什么需要清洗

### 1.1 `Gold=0` 不是没有修改

表格里的 `Gold Module=0` 或 `Gold Function=0` 容易被误解。它不是说这个样本没有修改文件，也不是说 gold patch 为空。

它的实际含义是：**真实 patch 的修改行没有落入当前 `repo_structures` 抽取出的 module/function 实体范围。**

常见原因包括：

- 修改的是 `import`、顶层常量、配置项、测试数据、快照、JSON/YAML/Markdown、样式文件等非函数区域。
- 修改的是文件级脚本代码，不在任何函数或类中。
- 语言或语法结构比较特殊，结构抽取器没有识别出函数、类、方法。
- `repo_structures` 缺失、不完整，或函数起止行范围不准。

因此，这类样本仍然可以用于 **file-level** 评估，但不适合作为 module/function 主指标样本。否则 module/function 指标会同时测到两件事：模型定位能力和结构抽取覆盖率。

### 1.2 `Gold>15` 会让 Acc@15 变成理论不可能

我们的 `Acc@K` 是严格 full-coverage：只有 top-k 预测覆盖全部 gold 目标时才记为 1。

如果一个样本有 22 个真实函数需要修改，那么即使模型把前 15 个预测全都预测对，`Function Acc@15` 也不可能为 1，因为 top15 最多只能覆盖 15 个目标。这不是模型失败，而是指标和样本难度之间存在上限冲突。

这类样本仍然有价值，尤其适合观察模型是否能处理大 patch、多文件、多函数长尾问题；但它们更适合作为 **long-tail diagnostic set**，配合 `Set Metrics @All`、Recall、Precision、F1 或不截断指标看，而不是混进主 `Acc@15` 排名。

## 2. 建议的清洗口径

这里建议把样本分成三类，而不是简单一刀切：

| 集合 | 过滤条件 | 适合用途 |
|---|---|---|
| File-clean@15 | `1 <= Gold File <= 15` | file-level 主评估 |
| Entity-clean@15 | `1 <= Gold Module <= 15` 且 `1 <= Gold Function <= 15` | module/function 主评估 |
| Three-level-clean@15 | file、module、function 三层都在 `1..15` | 三层横向主评估，最推荐 |
| Long-tail / Unmappable diagnostic | 任一层 `>15`，或 module/function 为 0 | 诊断真实长尾、结构抽取覆盖、非函数修改 |

如果论文或报告只需要一个主表，建议用 `Three-level-clean@15` 做主评估，然后把原始全集结果作为“未清洗真实分布结果”附录报告。这样主表更公平，附录保留真实世界难度。

最明显的问题在 OmniGIRL：file-level 基本都可评估，但 module/function 可评估样本明显减少。尤其 OmniGIRL unified60 中有 20/60 个样本的 module/function gold 为 0，所以如果强行做三层统一主评估，只剩 40 个样本真正适合。

| 数据集 | File >15 | Module =0 | Module >15 | Function =0 | Function >15 |
|---|---:|---:|---:|---:|---:|

这里可以看出两个 benchmark 的问题类型不同：

## 5. 清洗后的 benchmark 该如何使用

### 5.1 主表建议

建议主实验表使用 `Three-level-clean@15`：

| Benchmark | 推荐主评估样本数 | 用途 |
|---|---:|---|
| SWE-bench Multimodal dev clean | 92 | SWE 全量三层主评估 |
| SWE-bench Multimodal 60 clean | 55 | SWE60 快速横向比较 |
| OmniGIRL full-candidates clean | 445 | Omni 候选全集三层主评估 |
| OmniGIRL unified60 clean | 40 | Omni60 三层快速横向比较 |

这样可以保证每个样本在 file、module、function 三层都有非空 gold，并且每层 gold 数量不超过 15。此时 `Acc@15` 才是“模型是否能在 top15 覆盖全部目标”的指标，而不是被样本天然上限卡死。

### 5.2 附录/诊断建议

不建议把被清洗掉的样本直接丢弃。它们应该单独报告：

- `Entity-unmappable`：module/function 为 0 的样本，用来分析结构抽取覆盖和非函数修改。
- `Long-tail >15`：任一层 gold 数量超过 15 的样本，用来分析大 patch、多文件、多函数真实复杂任务。
- `File-only`：module/function 不可映射但 file 可评估的样本，用来报告 file-level 定位能力。

对于这些诊断集，可以优先看：

- `Set Metrics @All`
- `Recall@All`
- `Precision@All`
- `MAP@15`
- `MRR@15`
- 模型是否能找到关键入口文件，而不是是否覆盖全部实体。

## 6. 对现有结果解释的影响

如果不清洗，当前结果里的 module/function 指标会混入两类问题：

1. **真实实体目标不存在或不可映射。** 这会让 module/function 指标不再纯粹代表定位能力。
2. **真实目标超过 15。** 这会让 `Acc@15` 对部分样本天然不可能成功。

所以之后解释 baseline 时建议采用双表：

1. 原始全集表：展示真实 benchmark 难度，保留所有长尾和不可映射问题。
2. Clean@15 主表：展示公平的三层定位能力。

这样讲故事会更清楚：原始全集回答“现实任务有多混乱”，clean 集回答“在三层 gold 定义清晰且 Acc@15 可达的样本上，模型到底定位得怎么样”。

## 7. 后续建议

建议后续新增一个固定清洗脚本，例如：

```bash
python notes/main/tupian/build_three_level_clean_benchmark.py \
  --samples LocAgent/newtest/omnigirl-unified60/data/samples.jsonl \
  --structures LocAgent/newtest/omnigirl-unified60/repo_structures \
  --output LocAgent/newtest/omnigirl-unified60-clean15/data/samples.jsonl \
  --excluded LocAgent/newtest/omnigirl-unified60-clean15/excluded_samples.jsonl \
  --manifest LocAgent/newtest/omnigirl-unified60-clean15/manifest.json
```

脚本应输出：

- `clean samples.jsonl`：满足 `1 <= file/module/function <= 15` 的样本。
- `excluded_samples.jsonl`：被排除样本及原因。
- `manifest.json`：原始数量、清洗数量、每类原因数量、数据路径、生成时间。

这样之后所有 baseline 都可以在同一个 clean benchmark 上跑，避免每次手工解释哪些样本不适合三层评估。
