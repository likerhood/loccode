# 已有定位结果的 Clean15 复评估方案与脚本说明

这份文档说明一个新的评估流程：**不重新运行任何 baseline，只基于已经生成的定位结果，筛选出 clean15 样本子集，然后重新计算 file/module/function 三层指标。**

这样做的目的很直接：之前所有 baseline 已经跑过完整 benchmark 或 60 子集，如果现在为了更公平的样本口径重新跑一遍，成本太高，而且还会受到接口波动、额度、随机性和环境差异影响。Clean15 复评估只改变评估样本集合，不改变 baseline 预测本身，因此更适合做补充分析。

## 1. Clean15 是什么

Clean15 默认使用 `three-level` 口径：

```text
1 <= gold file 数 <= 15
1 <= gold module 数 <= 15
1 <= gold function 数 <= 15
```

也就是说，一个样本要被纳入 clean15，需要同时满足：

- file-level 有可评估 gold，且真实修改文件数不超过 15；
- module-level 有可映射 gold，且真实 module 数不超过 15；
- function-level 有可映射 gold，且真实 function 数不超过 15。

这样可以避免两类问题：

1. `Acc@15` 本身只看前 15 个预测，但有些样本 gold function 数超过 15，这种样本即使模型给出很多正确函数，也很难在 strict full-coverage 指标里公平比较。
2. 有些样本真实改了文件，但修改落在配置、快照、顶层常量、import 或结构抽取不到的位置，导致 module/function gold 为 0。这类样本适合 file-level 评估，但不适合三层统一评估。

## 2. 新增脚本

### 2.1 生成 clean 子集

脚本：

```text
scripts/build_three_level_clean_subset.py
```

功能：

- 输入原始 `samples.jsonl` 和对应 `repo_structures/`；
- 按 gold patch 计算每个样本的 gold file/module/function 数；
- 输出 clean samples、clean IDs、逐样本诊断、manifest；
- 默认 `--mode three-level --max-gold 15`。

输出文件示例：

```text
clean_subsets/omnigirl-unified60.clean15.samples.jsonl
clean_subsets/omnigirl-unified60.clean15.ids.txt
clean_subsets/omnigirl-unified60.clean15.per_sample.jsonl
clean_subsets/omnigirl-unified60.clean15.excluded.jsonl
clean_subsets/omnigirl-unified60.clean15.manifest.json
```

### 2.2 复评估单个结果目录

脚本：

```text
scripts/reevaluate_existing_results_clean15.py
```

功能：

- 输入 clean samples、已有预测文件、结构目录、结果目录；
- 可选生成过滤后的预测文件；
- 写入新的评估目录：

```text
<result-dir>/eval_clean15/metrics_3level.md
<result-dir>/eval_strict_clean15/metrics_3level.md
```

默认使用 fast 三层评估器：

```text
scripts/eval_3level_localization_fast.py
```

这个 fast 版本只解析 gold 修改文件和预测文件，不展开整个仓库结构，所以适合批量复评估。若要完全复用原始 LocAgent 三层评估脚本，可以加：

```bash
--eval-backend official
```

### 2.3 批量复评估所有 baseline

脚本：

```text
scripts/reevaluate_all_existing_results_clean15.sh
```

功能：

- 自动生成 clean subset；
- 自动查找常见 baseline 结果目录；
- 批量复评估 LocAgent / CoSIL / GraphLocator / GALA / MM-IR；
- 不存在的预测文件会自动跳过。

## 4. 单个 baseline 手动复评估示例

如果只想对 GraphLocator 的 Omni60 结果复评估：

```bash
cd /home/like/locCode

python scripts/build_three_level_clean_subset.py \
  --samples LocAgent/newtest/omnigirl-unified60/data/samples.jsonl \
  --structure-dir LocAgent/newtest/omnigirl-unified60/repo_structures \
  --output-prefix clean_subsets/omnigirl-unified60.clean15 \
  --mode three-level \
  --max-gold 15 \
  --write-diagnostic

CONDA_ENV_ROOT=/home/like/miniconda3/envs \
python scripts/reevaluate_existing_results_clean15.py \
  --samples clean_subsets/omnigirl-unified60.clean15.samples.jsonl \
  --pred-file GraphLocator/newtest/omnigirl-unified60/results/openai_qwen3-vl-8b/loc_results.json \
  --structure-dir LocAgent/newtest/omnigirl-unified60/repo_structures \
  --result-dir GraphLocator/newtest/omnigirl-unified60/results/openai_qwen3-vl-8b \
  --python /home/like/miniconda3/envs/locagent/bin/python \
  --force
```

## 5. 输出怎么看

复评估不会覆盖原始指标：

```text
eval/metrics_3level.md
eval_strict/metrics_3level.md
```

它只新增：

```text
eval_clean15/metrics_3level.md
eval_strict_clean15/metrics_3level.md
filtered_predictions_clean15/
```

因此后续写论文或报告时，可以并排比较：

- 原始全集指标：反映完整 benchmark 难度；
- clean15 指标：反映三层 gold 都可映射、且 `Acc@15` 上限合理的子集表现；
- excluded 诊断：解释哪些样本因为 module/function 不可映射或 gold 数过大被排除。
