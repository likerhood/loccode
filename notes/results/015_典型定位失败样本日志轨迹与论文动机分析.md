# 典型定位失败样本、日志轨迹与论文动机分析

本文档整理 baseline 在多模态、跨语言 Issue 定位中的典型失败样本。目标不是重复指标表，而是把“为什么失败”讲清楚：Issue 里有哪些图片、URL、代码线索，baseline 实际搜了什么、预测了什么，和真实 gold 之间差在哪里。后续写论文动机、方法设计和案例分析时，可以直接引用这些失败模式。

本文使用三个判断：

- `file_rec@all`：预测文件集合里覆盖了多少真实 gold 文件。
- `module_rec@all`：预测 module 集合里覆盖了多少真实 gold module。
- `function_rec@all`：预测 function 集合里覆盖了多少真实 gold function。

如果一个样本多个 baseline 都是 `file_rec@all=0`，说明它不是单个方法偶然失误，而更可能暴露了 benchmark 证据结构和当前方法范式之间的系统性断裂。

## 2. 失败模式总览

| 失败模式 | 典型样本 | 主要现象 | 对论文动机的价值 |
|---|---|---|---|
| URL 指向相关代码，但不是最终修改点 | `Automattic__wp-calypso-21409` | 模型强跟随 issue 里的 GitHub URL，预测 `current-user/selectors.js` 或 signup 文件，但真实修改在 WooCommerce dashboard/address 流程 | 需要区分“证据 URL”和“修复目标 URL”，不能把 URL 直接当 gold seed |
| 路由 URL 没有解析到页面组件树 | `Automattic__wp-calypso-21648` | Issue 给出 `/store/settings/email/{DOMAIN}`，baseline 找到 site title、decode entities、通知组件，但没有定位到 WooCommerce email settings 组件 | 需要 route-to-component、route-to-state 的结构化解析 |
| 图片描述 UI 现象，但没有转换成代码约束 | `Automattic__wp-calypso-22026` | 截图显示 loading indicator spacing 问题，模型找 spinner、store address 或泛化 style 文件，真实修改跨 Woo settings/dashboard/style/state | 需要视觉证据转成 UI region、组件、样式属性、状态流 |
| 外部运行 URL 隐含数据模型和 URL 生成逻辑 | `Automattic__wp-calypso-23915` | Reader edit link 失败，baseline 找 Reader UI、post action、sidebar，但真实函数是 `getEditURL` | 需要从用户操作 URL 推断 URL builder/normalizer |
| 依赖/许可证/构建配置类问题跨文件闭包大 | `Automattic__wp-calypso-33948` | 模型找到 localforage 或 package 相关文件，但无法覆盖 browser-storage、initial-state、webpack、package/shrinkwrap 的完整闭包 | 需要 patch closure，不只是 top-k 相关文件 |
| 图片/示例 URL 指向复现代码，真实修复在底层语义层 | `diegomura__react-pdf-1178`、`1280` | 模型追到 examples、renderer、layout node，但 gold 在 stylesheet expansion/resolve | 需要从视觉布局症状反推样式解析语义层 |
| 渲染结果异常指向插件表象，真实问题在生命周期/缓存 | `chartjs__Chart.js-8162` | legend/title rendered twice，baseline 都找 legend/title plugin，gold 是 `core.plugins.js::invalidate` | 需要区分“受影响组件”和“生命周期控制点” |
| 跨语言 JS + shader + WebGL 闭包只召回部分文件 | `processing__p5.js-5917` | baseline 能找到 `RendererGL.js` 和 `material.js`，但漏掉多个 `.frag` shader 文件，function 召回低 | 需要跨语言实体图和文件类型路由 |

## 3. 案例一：URL 是证据，不一定是修复目标

### 样本：`Automattic__wp-calypso-21409`

Issue 摘要：Store signup flow 需要在用户进入 `wp-admin` 前强制 email verification。Issue 包含两个 UI 截图，并给出一个 GitHub URL：

```text
https://github.com/Automattic/wp-calypso/blob/master/client/state/current-user/selectors.js#L157
```

真实 gold 文件集中在 WooCommerce dashboard/address flow：

```text
client/extensions/woocommerce/app/dashboard/index.js
client/extensions/woocommerce/app/dashboard/required-plugins-install-view.js
client/extensions/woocommerce/app/dashboard/setup-footer.js
client/extensions/woocommerce/app/dashboard/store-location-setup-view.js
client/extensions/woocommerce/components/address-view/index.js
client/extensions/woocommerce/components/form-location-select/states.js
client/extensions/woocommerce/components/query-locations/index.js
client/extensions/woocommerce/lib/countries/index.js
client/extensions/woocommerce/state/sites/locations/selectors.js
client/extensions/woocommerce/state/sites/settings/actions.js
```

Clean15 严格评估结果：

| Baseline | file_rec@all | module_rec@all | function_rec@all | 预测倾向 |
|---|---:|---:|---:|---|
| LocAgent | 0.00 | 0.00 | 0.00 | `current-user/selectors.js`、theme/ui state、server loader |
| CoSIL | 0.00 | 0.00 | 0.00 | signup step 文件 + `current-user/selectors.js` |
| GraphLocator | 0.00 | 0.00 | 0.00 | `client/jetpack-connect/signup.js` |
| GALA | 0.00 | 0.00 | 0.00 | signup step / plans 文件 |
| BM25-MMIR | 0.10 | 0.20 | 0.10 | 命中 `store-location-setup-view.js`，但覆盖不足 |

LocAgent 轨迹片段显示，它被 issue 中的函数名和 URL 明确牵引：

```text
ASSISTANT: search_code_snippets(["isCurrentUserEmailVerified"])
TOOL: auto-filled search terms from issue context:
['wp-admin', 'isCurrentUserEmailVerified', '/wp-admin',
 'user-images.githubusercontent.com/...png', ...]
ASSISTANT: search_code_snippets(["isCurrentUserEmailVerified"],
 file_path_or_pattern="client/state/current-user/selectors.js")
```

LocAgent 最终输出：

```text
client/state/current-user/selectors.js
client/state/themes/selectors.js
client/state/ui/selectors.js
client/state/themes/utils.js
client/state/ui/guided-tours/contexts.js
client/state/utils.js
client/state/ui/editor/image-editor/actions.js
server/pages/index.js
server/bundler/loader.js
```

CoSIL 的 LLM 输出也呈现同样偏移：

```text
client/signup/steps/site.js
client/signup/steps/plans.js
client/signup/steps/design-type-with-store.js
client/signup/steps/site-or-domain.js
client/state/current-user/selectors.js
```

### 失败原因

这个样本的 URL 是有效证据，但它只是“当前用户 email 是否验证”的状态入口，不是最终 patch 所在位置。真正需要修改的是 WooCommerce signup/dashboard/address flow 如何使用这个状态、如何阻断流程、如何处理地址页与 store location setup。

当前 baseline 普遍缺少一个判断：**URL 指向的文件是 evidence seed，还是 patch target？** 它们把 evidence seed 当成 target seed，导致搜索图从一开始就偏了。

### 对方法设计的启发

需要引入“证据角色识别”：

- `code-url/evidence-only`：Issue 里提到的代码 URL 只是上下文或状态来源。
- `route-url/page-entry`：运行 URL 指向页面入口。
- `image/ui-symptom`：截图描述 UI 状态。
- `patch-target`：真正需要修改的文件/函数。

Agent 不应直接把 evidence URL 排到前面，而应先构造 evidence graph：`email verified selector -> signup/dashboard flow -> WooCommerce address setup -> block/redirect behavior`，再从图中选择 patch target。

## 4. 案例二：路由 URL 没有被解析成页面组件

### 样本：`Automattic__wp-calypso-21648`

Issue 摘要：Store email settings 的 From Name placeholder 显示未解码实体。Issue 明确给出运行路径：

```text
http://calypso.localhost:3000/store/settings/email/{DOMAIN}
```

真实 gold 文件：

```text
client/extensions/woocommerce/app/settings/email/email-settings/components/customer-notification.js
client/extensions/woocommerce/app/settings/email/email-settings/components/internal-notification.js
client/extensions/woocommerce/app/settings/email/email-settings/components/notifications-origin.js
client/extensions/woocommerce/app/settings/email/email-settings/index.js
client/extensions/woocommerce/app/settings/email/email-settings/style.scss
client/extensions/woocommerce/state/sites/settings/email/actions.js
client/extensions/woocommerce/state/sites/settings/email/reducer.js
```

Baseline 预测与指标：

| Baseline | file_rec@all | 典型预测 |
|---|---:|---|
| LocAgent | 0.00 | `mailchimp/setup-steps/store-info.js`、account recovery、notices |
| CoSIL | 0.00 | `site-title`、`decode-entities`、`email-verification`、`site-selector` |
| GraphLocator | 0.00 | `woocommerce-services/views/label-settings` |
| GALA | 0.00 | `site-settings/actions.js`、`decode-entities/browser.js`、site settings reducer |
| BM25-MMIR | 0.00 | title parser、media plugin、post normalizer 等泛化命中 |

LocAgent 轨迹片段：

```text
ASSISTANT:
The problem describes an issue where an HTML entity (`&#039;`) is not being decoded
in the "From name" field placeholder or value in the email settings page...
Search for files related to email settings and the "From name" field.

TOOL:
auto-filled search terms:
['&#039;', 'site_title', '//calypso.localhost:3000/store/settings/email/{DOMAIN}',
 'store', 'settings', 'email', '{DOMAIN}']
```

CoSIL 输出：

```text
client/components/site-title/
client/lib/formatting/decode-entities/
client/components/email-settings/
client/lib/i18n-utils/
client/components/site-settings/
```

### 失败原因

模型理解到了“HTML entity decoding”和“email settings”，但没有把 route `/store/settings/email/{DOMAIN}` 解析到 WooCommerce extension 下的 email settings 页面。它选择了更通用的 site title、formatting、site settings，而真实代码在：

```text
client/extensions/woocommerce/app/settings/email/email-settings/
```

也就是说，文本语义是对的，但仓库结构解析失败了。

### 对方法设计的启发

多模态 Issue 定位不能只做文本检索，需要增加 route resolver：

1. 从 URL 解析出 route tokens：`store / settings / email / {DOMAIN}`。
2. 在仓库中检索 route registration、navigation、controller、page component。
3. 把 route path 映射到 feature root：`client/extensions/woocommerce/app/settings/email/`。
4. 再在 feature root 内做 field-level/entity-level rerank。

这类样本可以作为论文中“URL 是结构化证据，但当前 baseline 只把它当普通字符串”的核心案例。

Issue 摘要：shipping page loading 时，address loading indicator 和 unit inputs 之间缺少 spacing。Issue 主要证据是一张 UI 截图。

真实 gold 涉及多个 WooCommerce settings/dashboard/style/state 文件：

```text
client/extensions/woocommerce/app/dashboard/setup-tasks.js
client/extensions/woocommerce/app/settings/email/mailchimp/sync_tab.js

client/extensions/woocommerce/app/settings/taxes/taxes-options.js
client/extensions/woocommerce/app/settings/taxes/taxes-wcs.js
client/extensions/woocommerce/state/sites/settings/general/...
```

Baseline 预测：

| Baseline | file_rec@all | 典型预测 |
|---|---:|---|
| LocAgent | 0.00 | theme/ui selectors、image-editor actions、server pages |
| CoSIL | 0.00 | async-load、spinner、forms、sorted-grid style |
| GraphLocator | 0.00 | `shipping-origin.js` |
| GALA | 0.00 | store-address、form-dimensions-input、store-sidebar |
| BM25-MMIR | 0.00 | generic media/style/state 文件 |

### 失败原因

这个样本的文本中有 “loading indicator”“unit inputs”“spacing”，模型容易检索到 spinner、form input、store address 这类表面组件。但是 gold 不只是单个 spinner 组件，而是 WooCommerce settings 多页面样式与状态逻辑的 patch closure。

```text

视觉问题：loading placeholder 与 unit input 间距不足
可能代码类型：SCSS / layout style / form row component / loading state branch
```

### 对方法设计的启发

需要视觉证据抽取器输出“可检索约束”，而不是只把图片 URL 放进 prompt：

- UI 页面类型：dashboard、settings、reader、chart、PDF render、canvas/webgl。
- 视觉异常类型：spacing、overlap、duplicate render、color/alpha、alignment、tooltip/click region。
- 代码搜索类型：style 文件、layout engine、render lifecycle、event hit testing、state selector。
- 与 route/feature root 联合约束：`woocommerce + settings + shipping + style.scss`。

## 6. 案例四：外部运行 URL 隐含 URL 生成函数

### 样本：`Automattic__wp-calypso-23915`

Issue 摘要：Reader 里自己 Jetpack site 的 post Edit link 失效。Issue 给出运行 URL：

```text
https://wordpress.com/read/list/goldsounds/amptest
https://wordpress.com/read/feeds/76552105/posts/1804969891
```

评估结果和模型推理中指向的 WebGL/shader 闭包：

```text
client/lib/posts/utils.js::getEditURL
```

Baseline 预测：

| Baseline | file_rec@all | 典型预测 |
|---|---:|---|
| LocAgent | 0.00 | reader list、reader reducer/actions/sidebar/controller |
| CoSIL | 0.00 | post editor action bar、post edit button、reader posts/actions |
| GraphLocator | 0.00 | post-type-list edit menu |
| GALA | 0.00 | reader post actions、reader full post style |
| BM25-MMIR | 0.00 | media plugin、sharing examples、guided tours |

### 失败原因

所有方法都意识到这是 Reader/Edit link 问题，但都停在 UI action 或 Reader 页面层，没有进一步追踪“Edit link 是如何构造出来的”。真实修复点是 URL builder：

```text
client/lib/posts/utils.js::getEditURL
```

这说明当前 agent 在“用户点击路径 -> UI action -> URL generator -> site/post identity normalization”的链条上断掉了。

### 对方法设计的启发

这类样本要求 agent 具备行为链推理：

```text
Reader post page
  -> Edit link element
  -> click target URL
  -> post/site metadata
  -> getEditURL / URL builder
```

论文可以把它作为“用户级 reproduction steps 需要转成程序级 dataflow”的案例。

## 7. 案例五：依赖替换和构建配置需要多文件 patch closure

### 样本：`Automattic__wp-calypso-33948`

Issue 摘要：`localForage` 使用 Apache 2.0 license，与 GPLv2 不兼容，需要替换/移除相关依赖。Issue 有截图，但核心是依赖和 license。

真实 gold 文件：

```text
CREDITS.md
client/lib/browser-storage/README.md
client/lib/browser-storage/bypass.ts
client/lib/browser-storage/index.ts
client/lib/user/support-user-interop.js
client/lib/user/user.js
client/state/initial-state.js
npm-shrinkwrap.json
package.json
webpack.config.js
```

Baseline 预测：

| Baseline | file_rec@all | 典型预测 |
|---|---:|---|
| LocAgent | 0.00 | 空预测 |
| CoSIL | 0.00 | `client/lib/localforage/index.js`、local-storage-polyfill、feature-detection |
| GraphLocator | 0.00 | notifications selectors、jetpack remote install selectors、package |
| GALA | 0.00 | apps package、webpack config 类文件 |
| BM25-MMIR | 0.20 | 命中 `package.json`、`webpack.config.js` 等少数文件 |

### 失败原因

这个样本不是“找一个 bug 函数”，而是“替换一个依赖带来的全局存储抽象和构建闭包”。CoSIL 找到了 `localforage` 表面文件，但没有推断出需要引入新的 `browser-storage` abstraction，并同步修改 initial state、user storage、package、shrinkwrap、webpack。

这是 top-k file localization 的典型边界：如果 gold patch 是依赖迁移或架构替换，单点检索很难覆盖完整 patch closure。

### 对方法设计的启发

需要显式识别 patch 类型：

- bug fix：优先定位错误函数。
- UI style fix：优先定位组件 + style。
- dependency/license migration：优先构造 dependency closure。
- API behavior change：优先构造 call/dataflow closure。

对 dependency/license migration，agent 应该从 `localForage` 这个 seed 扩展：

```text
dependency declaration -> import/use sites -> storage abstraction -> runtime state init -> build config -> lockfile/docs
```

### 失败原因

### 对方法设计的启发

对 PDF/layout benchmark，agent 应该先识别症状类型：

```text
visual diff: margin auto / flexBasis / stretch / color alpha
semantic layer: stylesheet expansion / unit resolve / layout engine / renderer
```

然后把 example URL 降权为 reproduction evidence，把底层 style pipeline 升权。

## 13. 从失败日志抽象出的共性问题

### 13.1 当前 agent 的搜索逻辑太容易被表面词牵引

LocAgent 和 CoSIL 的轨迹中反复出现一种模式：

```text
Issue mentions "legend/title" -> search plugin.legend/plugin.title
Issue mentions "margin auto" + example URL -> search examples/knobs
Issue mentions "email decode" -> search site-title/decode-entities
Issue contains GitHub code URL -> search that exact file
```

这种逻辑在简单样本上有效，但在多模态 Issue 中经常不够。图片和 URL 往往是 evidence，而真实 patch 在更深的机制层。

### 13.2 图方法也依赖起点，起点错了图扩展会继续错

GraphLocator 和 GALA 有图或结构信息，但它们同样会从错误 seed 出发：

- `21409` 从 signup/jetpack connect 出发，没有进入 WooCommerce dashboard/address flow。
- `8162` 从 legend/title plugin 出发，没有进入 `core.plugins.js` lifecycle。
- `1178` 从 example/renderer/layout node 出发，没有进入 stylesheet expand/resolve。

因此问题不是“有没有图”，而是“图的 seed 是否经过证据角色判断和机制假设校验”。

### 13.3 多模态信息需要被翻译成可执行检索约束

图片 URL 本身没有定位能力。它必须先变成结构化语义：

```text
图片看到：legend/title duplicated
机制假设：plugin redraw lifecycle / cache invalidation
检索目标：core.plugins / controller update lifecycle
```

或者：

```text
图片看到：spacing between loading indicator and unit inputs
机制假设：style/layout branch for loading state
检索目标：feature root + SCSS + form row component + loading state
```

当前 baseline 大多没有这一层。

### 13.4 Clean15 仍然包含多文件、多函数、跨资源闭包

Clean15 去掉了超过 15 个 gold 的极端样本，但并不等于样本都简单。`33948` 有 10 个 gold 文件；`5917` 属于 WebGL/资源闭包问题；`21409` 有 10 个 WooCommerce 文件。Clean15 更适合公平评估 top-15，但仍然要求 agent 有 patch closure 能力。

## 14. 可转化为论文动机的核心论断

可以把上述案例压缩成以下论文动机：

1. **多模态 Issue 中，图片和 URL 经常是证据入口，不是修复目标。** 现有 baseline 容易把 URL 指向文件、截图表面组件、复现 example 当成 patch target，导致定位偏移。
2. **跨语言仓库中，真实修复常发生在“机制层”而不是“现象层”。** 例如 Chart.js 的 duplicated title/legend 表象在 plugin 层，但修复在 lifecycle invalidation；react-pdf 的 visual diff 表象在 example/rendering，修复在 stylesheet expansion。
3. **图检索的关键不是图本身，而是 seed 选择和机制约束。** 现有图方法从错误 seed 扩展，只会稳定地产生相邻但错误的候选。
4. **Clean15 之后仍然需要多文件 patch closure。** 依赖迁移、WebGL/shader、WooCommerce flow 这类样本要求覆盖配置、样式、状态、资源文件和函数实体的组合。

## 15. 对我们框架的具体要求

基于上述失败案例，一个更强的多模态跨语言 agent 定位框架至少需要以下模块。

### 15.1 Evidence Role Classifier

先判断每条证据的角色：

| 证据 | 可能角色 | 处理方式 |
|---|---|---|
| GitHub code URL | reproduction/evidence/seed | 不直接当 gold，检查引用位置是否只是状态来源 |
| localhost/wordpress.com route | page-entry | 解析 route 到 controller/component/root directory |
| user-images 截图 | visual symptom | 转换成 UI/布局/渲染/颜色/交互约束 |
| external issue URL | related symptom | 抽取共性机制，不直接扩展到外部文本 |
| example/repl URL | reproduction code | 降权为复现入口，追踪到底层 parser/layout/render pipeline |

### 15.2 Mechanism Hypothesis Generator

从表象生成机制假设：

```text
legend/title rendered twice -> plugin lifecycle / cache invalidation
margin auto broken -> stylesheet expand/resolve / layout engine
edit link broken -> URL builder / post metadata / site type
spacing wrong -> SCSS/layout branch / loading state
license incompatible -> dependency closure / storage abstraction / lockfile
```

### 15.3 Cross-language Artifact Graph

统一建模不同文件类型：

- JS/TS 函数、React 组件、Redux actions/reducers/selectors。
- SCSS/CSS 样式节点。
- GLSL shader 文件。
- Markdown/MDX 文档和 samples。
- package/lockfile/webpack/babel/tsconfig。
- route config、URL builder、data-layer API。

### 15.4 Failure-aware Verifier

在输出前验证候选是否覆盖关键证据链：

- 如果 issue 有 route，预测文件是否位于 route 对应 feature root？
- 如果 issue 是 visual/layout，预测是否包含 style/layout/render pipeline？
- 如果 issue 是 dependency/license，预测是否包含 package/lockfile/use-site closure？
- 如果 issue 提到 screenshot component，预测是否只停留在表象组件而没有机制文件？

## 16. 后续实验建议

为了把这些案例转成论文中的实验支撑，可以增加三个分析表：

1. **证据角色失败表**：每个失败样本标注 URL 是 `direct-code`、`route`、`external-page`、`image-only`、`example/repl`。
2. **表象层 vs 机制层表**：记录 baseline 预测是否停在表象层，gold 是否在机制层。
3. **patch closure 覆盖表**：对多文件 gold，统计 baseline 是否命中核心 seed、是否覆盖配置/样式/资源/状态闭包。

这些表可以把“baseline 为什么错”从个案描述变成系统性证据。
