# 证据感知多语言仓库异构图详细规划

本文基于前面关于“证据感知的多语言仓库异构图”的讨论，进一步给出可落地的工程规划。这里的图不是为了可视化，而是为了让 Agent 在多模态、多 URL、多语言仓库中更稳定地完成 issue localization。

核心目标是：

```text
把 issue 里的图片、URL、错误描述、API 名、代码片段转换成结构化证据节点；
把仓库里的文件、函数、类、路由、组件、配置、样式、文档、测试转换成语义节点；
把它们之间的关系建成 typed edges；
再让 Agent 沿这些边进行可解释的候选扩展和 patch closure 验证。
```

## 1. 问题定义

当前 baseline 的定位流程通常是：

```text
Issue 文本 / 图片 caption / URL 摘要
  -> BM25 / dense retrieval / LLM search
  -> 局部读文件
  -> 输出 top-k 文件或函数
```

这个流程在关键词与真实文件高度重合时有效，但在我们的 benchmark 中经常失效。主要原因不是模型完全看不到线索，而是线索角色被理解错了。

典型失败模式包括：

| 失败模式 | 具体表现 | 需要的图能力 |
|---|---|---|
| URL 误导 | URL 指向相关代码，但不是真实修复文件 | URL role classification + evidence-only 降权 |
| 图片弱映射 | 图片展示 UI 症状，但真实修复在 state/action/url builder | image -> UI text/route/component/state 的路径 |
| Playground 未解析 | playground URL 包含复现代码和配置，但被当普通网页 | playground decoder + parser/printer pipeline |
| 跨接口闭包缺失 | 只命中局部实现，漏掉 public API、backend、serializer | parameter flow + closure verifier |
| 多语言边缺失 | Python/Java/JS 的关键语义关系不同 | language router + language-specific edges |
| 非函数文件无法细评 | CSS/JSON/Markdown 没有 function，但仍是 gold patch | generalized semantic entity |

因此，我们需要的不是单一 call graph，而是：

```text
Evidence-aware Multilingual Semantic Repository Graph
证据感知的多语言语义仓库图
```

## 2. 设计原则

### 2.1 图必须有类型

不能只保存“节点 A 和节点 B 相邻”。必须明确边的语义：

```text
A imports B
A calls B
A renders B
A configures B
A styles B
A parameter_flows_to B
A delegates_to B
A is_evidence_for B
```

否则图扩展会变成无约束漫游，GraphLocator 这类方法已有类似问题：初始 seed 错或边类型弱时，图扩展只会扩大错误区域。

### 2.2 图必须区分证据和目标

Issue 里的 URL、图片、文档链接不一定是修复目标。它们可能只是 evidence seed。

例如：

```text
GitHub code URL -> selector 文件
真实 patch -> 使用该 selector 的 flow/component/address logic
```

因此 evidence node 必须有角色：

```text
direct_target
evidence_only
reproduction
behavior_spec
historical_patch
weak_background
```

### 2.3 多语言不是换 parser，而是换图关系

不同语言需要不同结构边：

```text
Python:
  symbol table / binder frame / type flow / parameter flow

Java:
  public API -> internal delegate / override / resource ownership

JavaScript/TypeScript:
  route -> component -> selector/action/reducer/style/url builder

CSS/JSON/MDX:
  selector/config key/doc section -> component/plugin/API
```

因此，图 schema 要统一，但 edge extractor 要按语言和生态分别实现。

### 2.4 图服务于 Agent 搜索，不只是离线分析

最终 Agent 需要的不是整张图，而是可调用工具：

```text
classify_evidence()
search_graph_nodes()
expand_graph()
trace_paths()
verify_patch_closure()
```

也就是说，图是 Agent 的“可查询导航地图”。

## 3. 图的总体结构

建议分成三层：

```text
Evidence Layer
  IssueText / Image / URL / Playground / Docs / GitHub issue/PR / Code URL

Repository Layer
  File / Module / Class / Function / Component / ConfigKey / StyleSelector / DocSection / Test

Semantic Relation Layer
  imports / calls / references / renders / selects_state / parameter_flows_to / delegates_to / styles / configures
```

整体流程：

```text
Issue Evidence
  -> Evidence Cards
  -> Seed Nodes
  -> Typed Expansion
  -> Candidate Files / Entities
  -> Closure Verification
  -> Ranked Localization Output + Path Explanation
```

## 6. Evidence Card 设计

每个样本先生成 evidence cards。

### 6.1 URL evidence card

字段：

```json
{
  "instance_id": "...",
  "url": "...",
  "url_type": "github_code | playground | docs | issue_pr | commit | external",
  "role": "direct_target | evidence_only | reproduction | behavior_spec | historical_patch | weak_background",
  "extracted": {
    "repo": "...",
    "file": "...",
    "line": 157,
    "symbols": ["isCurrentUserEmailVerified"],
    "api_names": [],
    "config": {}
  },
  "confidence": 0.86
}
```

URL role 的判断规则：

| URL 类型 | 默认角色 | 特殊处理 |
|---|---|---|
| GitHub blob + 文件在 gold 附近 | direct_target 候选 | 仍需 closure 扩展 |
| GitHub blob + 只提供状态/API 线索 | evidence_only | 降低直接 target 权重 |
| Playground | reproduction | 解码输入/配置 |
| Docs/API | behavior_spec | 抽 API、参数、行为规则 |
| GitHub issue/PR | historical_patch | 抽讨论、旧文件、相关 commit |
| 外部网页 | weak_background | 低权重 |

### 6.2 Image evidence card

字段：

```json
{
  "instance_id": "...",
  "image_url": "...",
  "image_type": "web_ui | chart_canvas | pdf_layout | markdown_render | diff_screenshot | unknown",
  "visible_text": ["Store Location", "Create your store"],
  "ui_elements": [
    {"type": "button", "text": "Create your store"},
    {"type": "input", "label": "Street address"}
  ],
  "symptom": "flow_not_blocked | layout_diff | chart_interaction | render_mismatch",
  "candidate_seeds": ["store location", "woocommerce", "email verification"]
}
```

图片不直接映射文件，而是先变成：

```text
visible text
route/page name
UI element
symptom type
```

再进入图扩展。


```text

```

## 9. Patch Closure Verifier

Patch closure 是这个图方法的关键。很多失败不是完全没命中，而是只命中一部分。

### 9.1 Closure 类型

| 类型 | 说明 | 示例 |
|---|---|---|
| API closure | public API、内部对象、backend、实现层都覆盖 | cryptography kdf_rounds |
| UI closure | route、component、state/action、utility/style 都覆盖 | wp-calypso store flow |
| Render closure | parser/resolve/layout/render/test 都覆盖 | react-pdf / chartjs |
| Config closure | config key、plugin/loader、consumer、test 都覆盖 | babel/prettier/webpack |
| Docs closure | docs section、API implementation、test/example 都覆盖 | docs/API 样本 |

### 9.2 Verifier 输出

```json
{
  "instance_id": "pyca__cryptography-7520",
  "candidate_files": ["..."],
  "closure_type": "api_parameter_flow",
  "covered_roles": ["public_api", "internal_object", "backend", "serializer"],
  "missing_roles": [],
  "score": 0.91,
  "explanation": "candidate set covers kdf_rounds propagation from public builder to OpenSSH serializer"
}
```

如果缺文件：

```json
{
  "missing_roles": ["public_api", "backend"],
  "suggested_expansion": ["parameter_flows_to", "api_to_backend"]
}
```

## 10. 和 Benchmark 的具体对应

### 10.1 SWE-bench Multimodal

SWE 重点处理多模态和前端/渲染链。

优先图路径：

```text
Image/Web UI:
  visible text
  -> route/page
  -> component
  -> state selector/action/reducer
  -> URL builder / style / utility

Chart/Canvas:
  option/API
  -> scale/legend/plugin
  -> event/render/update pipeline

PDF/Layout:
  style symptom
  -> stylesheet expand
  -> resolve styles
  -> layout engine
  -> renderer

Markdown/Text render:
  token/syntax
  -> parser
  -> renderer
  -> snapshot/test
```

典型样本：

```text

  code URL 是 selector evidence，不是 patch target。

diegomura__react-pdf-1178:
  example URL 是 reproduction evidence，真实 target 在 stylesheet expand/resolve。

  running URL 是 behavior evidence，真实 target 是 getEditURL 这类 URL builder。
```

### 10.2 OmniGIRL

Omni 重点处理 URL role 和 language router。

优先图路径：

```text
Python:
  docs/playground/API
  -> symbol table / binder / type flow / parameter flow

Java:
  public API / class name / docs
  -> delegate / override / implementation / resource ownership

JS/TS:
  playground/config
  -> parser / transform / printer / snapshot

C:
  symbol/function
  -> ownership / lifetime / error handling
```

典型样本：

```text

  del Foo -> TypeInfo -> binder.get_declaration。

  kdf_rounds -> public API -> backend -> OpenSSH serializer。

  SslHandler -> helper queue -> superclass override -> ByteBuf ownership operation。
```

## 11. 评估设计

### 11.1 定位指标

继续保留现有主指标：

```text
File Acc@1/3/5/8/10/12/13/15
File MRR@15
File MAP@15
File REC@All
```

实体级建议分两套：

```text
Function-level eligible:
  只在 gold function 可映射样本上算

Semantic entity-level:
  对 Function / StyleSelector / ConfigKey / DocSection / Fixture / Snapshot 等统一算
```

### 11.2 图方法新增诊断指标

```text
Evidence role accuracy:
  URL/图片角色是否判断正确

Seed hit rate:
  初始 evidence seed 是否命中 gold 邻域

Path hit rate:
  graph path 是否连接到 gold file/entity

Closure completeness:
  候选集合是否覆盖完整修复闭包

Expansion precision:
  图扩展后候选是否过度膨胀

Entity coverage:
  非函数实体抽取覆盖率
```

### 11.3 为什么要这些新指标

传统 Acc@K 只能告诉我们最后预测对不对，不能解释为什么错。

图方法需要回答：

```text
证据理解错了吗？
初始 seed 错了吗？
边扩展错了吗？
closure 漏了吗？
实体抽取没覆盖吗？
```

这能直接支撑论文里的 failure analysis。

## 13. 风险与边界

### 13.1 图过大

风险：

```text
全仓库全边构建成本高，Agent 查询容易过度扩展。
```

控制：

```text
按 instance 构建局部图；
只从 evidence seed 附近扩展；
限制 edge type 和 max hops；
对边设置 confidence。
```

### 13.2 边抽取不准

风险：

```text
heuristic edge 可能引入错误关系。
```

控制：

```text
保留 extractor/source/confidence；
Agent 使用高置信边优先；
低置信边只作为 recall 扩展；
closure verifier 做反向检查。
```

### 13.3 多模态证据解析不稳定

风险：

```text
图片 caption 或网页摘要不准。
```

控制：

```text
证据 card 中保存原始 URL/图片；
图片只作为 seed，不直接作为 target；
URL role 必须和 target rank 分离。
```

### 13.4 Function 指标和非函数实体混淆

风险：

```text
CSS/JSON/Markdown 被强行塞入 function-level，指标解释不严谨。
```

控制：

```text
Function eligible 单独报告；
Semantic entity-level 单独报告；
File-level 仍作为主指标。
```

## 15. 最终论文表述

可以把方法故事线写成：

```text
Existing localization agents often treat multimodal evidence as plain text and repository structure as files/functions.
However, our benchmark analysis shows that issue evidence frequently serves as reproduction or contextual evidence rather than direct patch targets, and patch locations are connected through language- and framework-specific semantic relations.

We therefore propose an evidence-aware multilingual semantic repository graph.
It unifies issue evidence, code entities, non-code repair entities, and language-specific semantic edges into typed traversable paths.
The agent performs evidence typing, graph-guided expansion, and patch closure verification, producing both ranked localization results and path explanations.
```

中文表述：

```text
我们不是把多语言仓库统一成普通调用图，而是构建证据感知的多语言语义仓库异构图。
该图把图片、URL、playground、文档、代码符号、配置、样式、路由、状态流和语言特定结构边统一成可搜索路径。
Agent 不再直接基于关键词猜文件，而是先识别证据角色，再沿 typed edges 扩展候选，最后验证候选集合是否形成完整修复闭包。
```

## 16. 结论

这套图规划的关键不是“图越大越好”，而是：

```text
证据要结构化；
边要有类型；
语言要有路由；
非函数实体要可评估；
Agent 搜索要能解释路径；
多文件 patch 要做 closure 验证。
```

如果只做 call graph，无法解决我们的 benchmark 中最核心的问题：URL 角色误判、图片到代码映射弱、跨接口参数闭包缺失、多语言结构边缺失、CSS/JSON/Markdown 非函数目标不可表达。

因此，下一步最实际的路线是先做 P0/P1 的轻量图：evidence card、基础 repo node、URL role、JS/Python/Java 高频语义边和 path explanation。这个版本就足以支撑论文动机、失败案例解释和 Clean15 结果上的 rerank/diagnostic 实验。
